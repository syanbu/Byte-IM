# B9.5 Server Delivery State Structures

This document explains the two server-side state structures used for receiver delivery ACK handling after B5.5.

## Overview

The mock server still uses two related in-memory structures inside `MessageRouter`:

1. `acceptedMessagesById`
2. `undeliveredMessagesByReceiver`

The difference from the earlier B9.5-only implementation is that these structures are no longer process-only state. They are now restored from SQLite during startup.

Persistent source:

- `mock-server/data/mock-im-messages.sqlite`

## 1. acceptedMessagesById

Definition:

```java
ConcurrentMap<String, AcceptedMessage> acceptedMessagesById
```

Key:

- `messageId`

Purpose:

- main server-side accepted-message record table
- primary lookup for sender duplicate-send idempotency
- current delivery-state snapshot for each accepted message

What it answers:

- has this `messageId` already been accepted?
- what was the original `MESSAGE_ACK`?
- what `serverSeq` was assigned?
- has the receiver already sent `DELIVERY_ACK`?

Persistence relationship:

- each in-memory record corresponds to one SQLite `accepted_messages` row
- on startup, `MessageRouter` loads persisted rows back into this map

## 2. undeliveredMessagesByReceiver

Definition:

```java
ConcurrentMap<String, ConcurrentMap<String, AcceptedMessage>> undeliveredMessagesByReceiver
```

Keys:

- first level: `receiverUserId`
- second level: `messageId`

Purpose:

- receiver-oriented secondary index over the undelivered subset
- targeted auth/reconnect replay without scanning the full accepted-message table

What it answers:

- which accepted messages for this receiver still lack delivery proof?

Persistence relationship:

- only rows with `delivered = 0` are restored into this structure at startup
- `DELIVERY_ACK` removes the message from this in-memory bucket and updates SQLite

## Relationship Between The Two Structures

These remain one main record table plus one secondary index:

- `acceptedMessagesById`: all accepted messages
- `undeliveredMessagesByReceiver`: only accepted messages still awaiting receiver delivery proof

Every undelivered message must exist in both places.
Delivered messages remain only in `acceptedMessagesById`.

## Message Lifecycle

### 1. Sender sends `SEND_MESSAGE`

The server:

- checks `acceptedMessagesById` for duplicate `messageId`
- allocates `serverSeq`
- builds sender `MESSAGE_ACK`
- persists the accepted message row in SQLite
- stores the record in `acceptedMessagesById`
- stores it in `undeliveredMessagesByReceiver[receiverUserId][messageId]`

### 2. Receiver is online

The server forwards `RECEIVE_MESSAGE`.

Until `DELIVERY_ACK` arrives, the message remains in both structures and in SQLite as `delivered = 0`.

### 3. Receiver is offline or fails before ACK

The message stays in:

- `acceptedMessagesById`
- `undeliveredMessagesByReceiver[receiverUserId]`
- SQLite `accepted_messages` with `delivered = 0`

### 4. Server restarts

On startup, `MessageRouter` loads persisted rows:

- all rows restore `acceptedMessagesById`
- only rows with `delivered = 0` restore `undeliveredMessagesByReceiver`

This is what makes sender idempotency and receiver replay restart-proof.

### 5. Receiver authenticates again

The server looks up:

```text
undeliveredMessagesByReceiver[receiverUserId]
```

and redelivers only that receiver's undelivered messages.

### 6. Receiver sends `DELIVERY_ACK`

The server:

- finds the accepted message by `messageId`
- verifies the `receiverUserId`
- marks the message delivered in SQLite
- removes the message from `undeliveredMessagesByReceiver[receiverUserId]`
- keeps the delivered accepted record in `acceptedMessagesById`

## Why Both Structures Still Matter

If only `acceptedMessagesById` existed:

- sender idempotency would work
- receiver replay would require scanning all accepted messages

If only `undeliveredMessagesByReceiver` existed:

- receiver replay would work
- sender duplicate-send idempotency would lose its direct `messageId` primary lookup

So the B5.5 design keeps the same in-memory roles while adding persistence underneath them.
