# B9.5 Server Delivery State Structures

This document explains the two in-memory data structures currently used by the mock server for B9.5 receiver delivery ACK handling.

Scope note:

- This describes the current mock-server implementation only.
- These structures are in-memory only.
- They are not durable across server restart.

## Overview

The current mock server keeps two related data structures for accepted chat messages:

1. `acceptedMessagesById`
2. `undeliveredMessagesByReceiver`

They serve different lookup paths and should be understood together.

## 1. acceptedMessagesById

Definition:

```java
ConcurrentMap<String, AcceptedMessage> acceptedMessagesById
```

Key:

- `messageId`

Purpose:

- This is the server-side main record table for already accepted messages.
- It is the primary structure for sender-side idempotency.
- It also stores the current delivery state for that accepted message.

Typical uses:

- When the sender retries the same `SEND_MESSAGE` with the same `messageId`, the server checks this map first.
- If the `messageId` already exists, the server treats it as an already accepted message.
- The server returns the original `MESSAGE_ACK`.
- The server does not allocate a new `serverSeq`.
- The server does not forward the message again as a fresh send.
- When the receiver later sends `DELIVERY_ACK`, the server uses `messageId` to find the accepted message record and update its delivery status.

What it answers:

- Has this `messageId` already been accepted by the server?
- What was the original `MESSAGE_ACK` for this message?
- What `serverSeq` was assigned?
- What is the current delivery state of this accepted message?

## 2. undeliveredMessagesByReceiver

Definition:

```java
ConcurrentMap<String, ConcurrentMap<String, AcceptedMessage>> undeliveredMessagesByReceiver
```

Keys:

- first-level key: `receiverUserId`
- second-level key: `messageId`

Purpose:

- This is a receiver-oriented secondary index for undelivered messages.
- It exists to support B9.5 redelivery on receiver auth/reconnect.
- It avoids scanning the whole accepted-message table for every authenticated user.

Typical uses:

- When a receiver authenticates, the server looks up only that receiver's bucket.
- The server redelivers the accepted messages in that receiver's undelivered bucket.
- When the server receives `DELIVERY_ACK`, it removes that message from the receiver's undelivered bucket.

What it answers:

- Which messages for this receiver still do not have delivery proof?

## Relationship Between The Two Structures

These are not two unrelated sources of truth.

They should be understood as:

- `acceptedMessagesById`: the main accepted-message record table
- `undeliveredMessagesByReceiver`: a receiver-based secondary index over the undelivered subset

In other words:

- every undelivered message should have a main record in `acceptedMessagesById`
- only messages still awaiting receiver delivery proof appear in `undeliveredMessagesByReceiver`

## Message Lifecycle

### 1. Sender sends `SEND_MESSAGE`

The server:

- accepts the message
- allocates `serverSeq`
- builds the sender `MESSAGE_ACK`
- creates an `AcceptedMessage`
- stores it in `acceptedMessagesById[messageId]`
- stores it in `undeliveredMessagesByReceiver[receiverUserId][messageId]`

### 2. Receiver is online

The server immediately forwards `RECEIVE_MESSAGE`.

If delivery proof is not yet received, the message remains in the receiver's undelivered bucket.

### 3. Receiver is offline or does not ACK

The message stays in:

- `acceptedMessagesById`
- `undeliveredMessagesByReceiver[receiverUserId]`

### 4. Receiver authenticates again

The server looks up:

```text
undeliveredMessagesByReceiver[receiverUserId]
```

and redelivers only messages in that receiver's bucket.

### 5. Receiver sends `DELIVERY_ACK`

The server:

- finds the main record through `acceptedMessagesById[messageId]`
- updates delivery state on that accepted message
- removes the message from `undeliveredMessagesByReceiver[receiverUserId]`

## Why Both Structures Are Needed

If only `acceptedMessagesById` existed:

- sender duplicate-send idempotency would still work
- but receiver auth/reconnect redelivery would require scanning all accepted messages and filtering by `receiverUserId`

If only `undeliveredMessagesByReceiver` existed:

- receiver redelivery lookup would work
- but sender duplicate-send idempotency would not have a direct primary lookup by `messageId`
- returning the original `MESSAGE_ACK` would be less direct

So the current design intentionally keeps:

- one primary lookup by `messageId`
- one secondary lookup by `receiverUserId`

## Current Non-Durable Limitation

Both structures are currently in-memory only.

That means:

- they live only inside the running mock-server process
- server restart clears both structures

After restart:

- sender-side accepted-message idempotency is lost for previously accepted messages
- receiver-side undelivered redelivery state is lost
- previously undelivered messages can no longer be redelivered from these structures alone

This is why restart-proof server delivery state is still a later persistence problem rather than a completed part of the current B9.5 in-memory scope.
