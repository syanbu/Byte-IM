# B9.5 服务端投递状态结构说明

本文说明 B5.5 之后，mock server 为接收侧 `DELIVERY_ACK` 使用的两类服务端状态结构。

## 概览

mock server 的 `MessageRouter` 里仍然使用两类相关的内存结构：

- `acceptedMessagesById`
- `undeliveredMessagesByReceiver`

和早期只做 B9.5 时不同的是，这两类结构现在不再只是进程内状态。mock server 启动时会从 SQLite 恢复它们，因此 sender-side `messageId` 幂等和 receiver-side 未投递消息重放可以跨 server 重启保留。

## 1. `acceptedMessagesById`

结构形态：

```text
ConcurrentMap<String, AcceptedMessage> acceptedMessagesById
```

key：

```text
messageId
```

value：

```text
AcceptedMessage
```

它表示服务端已经接受过的消息主记录。

主要用途：

- 判断 sender 重发同一个 `messageId` 时是否应返回原始 `MESSAGE_ACK`。
- 保留服务端分配过的 `serverSeq`。
- 保留原始消息 body，便于向 receiver 转发或重放。
- 保证服务端重启后不会把同一个 `messageId` 当成新消息重新分配序号。

持久化关系：

- SQLite `accepted_messages` 是跨重启的事实来源。
- 启动时，mock server 从 SQLite 加载 accepted rows，恢复到 `acceptedMessagesById`。
- 运行时，`acceptedMessagesById` 是快速查找索引。

## 2. `undeliveredMessagesByReceiver`

结构形态：

```text
ConcurrentMap<String, ConcurrentMap<String, AcceptedMessage>> undeliveredMessagesByReceiver
```

第一层 key：

```text
receiverUserId
```

第二层 key：

```text
messageId
```

value：

```text
AcceptedMessage
```

它表示每个 receiver 仍未完成 `DELIVERY_ACK` 的消息集合。

主要用途：

- receiver 在线时，消息转发后仍保留在该结构中，直到收到 `DELIVERY_ACK`。
- receiver 离线时，消息先留在该结构中，等 receiver 再次认证后重发。
- receiver 重新登录或 WebSocket 重连后，只重放属于该 receiver 的未投递消息。

持久化关系：

- SQLite `accepted_messages.delivered = 0` 的行会恢复到 `undeliveredMessagesByReceiver`。
- 收到 `DELIVERY_ACK` 后，server 同时从内存结构中移除该 receiver 的 message，并把 SQLite 中对应 delivery 标记为 delivered。

## 两个结构之间的关系

它们仍然是“一张主记录表 + 一个二级索引”的关系：

```text
acceptedMessagesById
    messageId -> AcceptedMessage

undeliveredMessagesByReceiver
    receiverId -> (messageId -> AcceptedMessage)
```

每条未投递消息必须同时存在于两个结构中：

- `acceptedMessagesById` 说明“服务端已经接受过这条消息”。
- `undeliveredMessagesByReceiver` 说明“这条消息还需要投递给某个 receiver”。

已投递完成的消息只需要保留在 `acceptedMessagesById` 中，用于 sender 重试幂等；不再需要留在 `undeliveredMessagesByReceiver` 中。

## 消息生命周期

### 1. Sender 发送 `SEND_MESSAGE`

服务端收到 sender 的 `SEND_MESSAGE` 后：

1. 检查 `acceptedMessagesById[messageId]`。
2. 如果已存在，返回原始 `MESSAGE_ACK`，不重新分配 `serverSeq`。
3. 如果不存在，分配新的 `serverSeq`。
4. 创建 `AcceptedMessage`。
5. 写入 SQLite `accepted_messages`。
6. 写入 `acceptedMessagesById`。
7. 写入 receiver 对应的 `undeliveredMessagesByReceiver`。

### 2. Receiver 在线

服务端向 receiver 转发 `RECEIVE_MESSAGE`。

在 receiver 发回 `DELIVERY_ACK` 之前，该消息仍然保留在两个内存结构中，并在 SQLite 中保持 `delivered = 0`。

### 3. Receiver 离线或 ACK 前失败

消息保留在：

```text
acceptedMessagesById[messageId]
undeliveredMessagesByReceiver[receiverId][messageId]
SQLite accepted_messages(delivered = 0)
```

这样 receiver 后续认证时可以重放。

### 4. Server 重启

启动时，`MessageRouter` 从 SQLite 加载 persisted rows：

- 所有 accepted rows 恢复到 `acceptedMessagesById`。
- `delivered = 0` 的 rows 恢复到 `undeliveredMessagesByReceiver`。

这使 sender 幂等和 receiver 重放都具备 restart-proof 能力。

### 5. Receiver 再次认证

server 查找：

```text
undeliveredMessagesByReceiver[receiverUserId]
```

然后只重发该 receiver 自己未完成 `DELIVERY_ACK` 的消息。

### 6. Receiver 发送 `DELIVERY_ACK`

server 收到 `DELIVERY_ACK` 后：

1. 校验 ack body 中的 `messageId`、`receiverId` 等字段。
2. 从 `undeliveredMessagesByReceiver[receiverId]` 移除该 `messageId`。
3. 更新 SQLite delivery 状态。
4. 保留 `acceptedMessagesById[messageId]`，用于 sender 后续重复发送时返回原始 ACK。

## 为什么两个结构都需要保留

如果只有 `acceptedMessagesById`：

- 可以判断 sender 重发幂等。
- 但 receiver 认证时无法快速知道“这个 receiver 还有哪些消息未投递”。

如果只有 `undeliveredMessagesByReceiver`：

- 可以做 receiver 重放。
- 但消息投递完成后就会消失，sender 重发同一个 `messageId` 时无法返回原始 `MESSAGE_ACK`。

因此 B5.5 保留原有两个内存角色，并在它们下方增加 SQLite 持久化事实来源。
