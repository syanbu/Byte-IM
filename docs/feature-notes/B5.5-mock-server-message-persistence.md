# B5.5 Mock-Server 消息持久化设计

## 目标

B5.5 的目标不是 Android 本地库，而是让 mock-server 在进程重启后仍然保留消息接受记录和 receiver 侧未完成 delivery 的状态。

这项能力解决的是以下问题：

- 服务端重启后，之前已接受的消息不能全部丢失
- sender 重发同一个 `messageId` 时，幂等不能因为服务端重启而失效
- receiver 没来得及 `DELIVERY_ACK` 的消息，不能因为服务端重启而永远丢失
- 后续 B4 服务端历史查询需要有一份可持续存在的消息基础表

## 范围

本次 B5.5 只覆盖 mock-server 侧 accepted-message 持久化与重启恢复。

明确不做：

- Android 本地数据库改造
- `HISTORY_QUERY` / `HISTORY_RESULT` 接口接入
- 新的已读回执语义
- B10-B13 扩展能力

## 与 B9 / B9.5 的关系

必须区分三件事：

- `MESSAGE_ACK`
  - 只表示服务端已接受消息并分配 `serverSeq`
  - 不表示 receiver 已收到

- `DELIVERY_ACK`
  - 表示 receiver 端已经持久化该消息
  - 不表示用户已读

- B5.5
  - 解决的是“服务端重启后，这些状态是否还能恢复”

所以 B5.5 不改变 B9/B9.5 语义，只是把原来进程内内存状态变成“可恢复状态”。

## 核心设计

### 运行时结构

`MessageRouter` 里仍然保留两个内存结构：

1. `acceptedMessagesById`
2. `undeliveredMessagesByReceiver`

### 两个变量的 key / value

#### 1. `acceptedMessagesById`

类型：

```java
ConcurrentMap<String, AcceptedMessage>
```

key：

- `messageId`

value：

- `AcceptedMessage`

含义：

- 这是服务端“已接受消息主表”的内存形态
- 每个 `messageId` 对应一条服务端已接受的消息记录
- 只要服务端认为这条消息已经 accepted，它就应该能在这里被找到

它解决的问题：

- sender 重发相同 `messageId` 时，服务端如何判断这是重复消息
- 服务端如何返回“第一次 accepted 时生成的原始 `MESSAGE_ACK`”
- 服务端如何知道这条消息的 `serverSeq`
- 服务端如何知道这条消息当前是否已经被 receiver `DELIVERY_ACK`

可以把它理解为：

```text
messageId -> AcceptedMessage
```

#### 2. `undeliveredMessagesByReceiver`

类型：

```java
ConcurrentMap<String, ConcurrentMap<String, AcceptedMessage>>
```

外层 key：

- `receiverId`

外层 value：

- 该 receiver 对应的“未完成 delivery 的消息桶”

内层 key：

- `messageId`

内层 value：

- `AcceptedMessage`

含义：

- 这是一个 receiver 维度的二级索引
- 它不是另一份独立事实来源，而是 accepted message 主表中“尚未 delivery 完成”的子集视图
- 只有 `delivered = false` 的消息才应该出现在这里

它解决的问题：

- receiver 重新 `AUTH` 后，服务端如何只查这个 receiver 自己还没 ACK 的消息
- 避免每次 receiver 上线都扫描整个 `acceptedMessagesById`
- 收到 `DELIVERY_ACK` 后，如何快速把该 receiver 的待补投消息移除

可以把它理解为：

```text
receiverId -> (messageId -> AcceptedMessage)
```

职责不变：

- `acceptedMessagesById`
  - 主记录表
  - 以 `messageId` 为键
  - 用于 sender duplicate send 幂等

- `undeliveredMessagesByReceiver`
  - receiver 维度的二级索引
  - 只保存还没有 `DELIVERY_ACK` 的消息
  - 用于 receiver `AUTH` 后定向补投

### 持久化来源

新增 SQLite 文件：

- `mock-server/data/mock-im-messages.sqlite`

其中 `accepted_messages` 表作为持久化来源。

内存不再是唯一事实来源。
SQLite 才是跨重启恢复的事实来源，内存更像运行时索引。

## 表结构

当前表结构如下：

```sql
CREATE TABLE IF NOT EXISTS accepted_messages (
  message_id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  sender_id TEXT NOT NULL,
  receiver_id TEXT NOT NULL,
  client_seq INTEGER NOT NULL,
  server_seq INTEGER NOT NULL,
  content TEXT NOT NULL,
  timestamp INTEGER NOT NULL,
  server_time INTEGER NOT NULL,
  delivered INTEGER NOT NULL DEFAULT 0,
  ack_json TEXT NOT NULL,
  message_json TEXT NOT NULL,
  UNIQUE(conversation_id, server_seq)
)
```

关键字段含义：

- `message_id`
  - sender 幂等主键

- `conversation_id`
  - 会话维度标识

- `client_seq`
  - sender 本地发送序号

- `server_seq`
  - conversation 维度的服务端权威序号

- `sender_id` / `receiver_id`
  - 路由信息

- `content`
  - 当前文本消息内容

- `timestamp`
  - sender 原始消息时间

- `server_time`
  - 服务端接受并生成 ACK 时的时间

- `delivered`
  - 是否已经收到 receiver `DELIVERY_ACK`

- `ack_json`
  - 原始 `MESSAGE_ACK` 协议体快照

- `message_json`
  - 原始 `RECEIVE_MESSAGE` 协议体快照

## 为什么只用一张表

当前没有单独建“undelivered 表”。

原因是：

- `acceptedMessagesById` 和 `undeliveredMessagesByReceiver` 不是两套独立事实来源
- 后者只是前者里 `delivered = 0` 的子集
- 用一张 accepted message 主表就足够表达：
  - 这条消息是否被服务端接受
  - 它属于哪个 receiver
  - 它是否已经 delivery 完成

启动时恢复逻辑是：

- 所有行恢复进 `acceptedMessagesById`
- 只有 `delivered = 0` 的行恢复进 `undeliveredMessagesByReceiver`

两者的关系可以总结为：

- `acceptedMessagesById`
  - 保存“全部已接受消息”

- `undeliveredMessagesByReceiver`
  - 保存“全部已接受消息里，尚未被 receiver delivery ACK 的那一部分”

也就是说：

- 每一条出现在 `undeliveredMessagesByReceiver` 里的消息，必然也存在于 `acceptedMessagesById`
- 但并不是每一条 `acceptedMessagesById` 里的消息都还会出现在 `undeliveredMessagesByReceiver`
- 一旦 receiver 回了 `DELIVERY_ACK`，这条消息仍会保留在 `acceptedMessagesById`，但会从 `undeliveredMessagesByReceiver` 中删除

## 写入与恢复流程

### 1. Sender 发送消息

当服务端收到 `SEND_MESSAGE`：

1. 解析 `messageId`、`conversationId`、`receiverId` 等字段
2. 先检查内存里的 `acceptedMessagesById`
3. 如果不是重复消息，则分配新的 `serverSeq`
4. 生成 `MESSAGE_ACK` 和带 `serverSeq` 的 `RECEIVE_MESSAGE` 体
5. 先把 accepted message 写入 SQLite
6. 再放入内存：
   - `acceptedMessagesById`
   - `undeliveredMessagesByReceiver[receiverId]`
7. 给 sender 回 `MESSAGE_ACK`
8. 如果 receiver 在线，尝试转发 `RECEIVE_MESSAGE`

关键点是：

- 不是定时器刷盘
- 不是异步批量落盘
- 而是在接受消息的链路上同步写入 SQLite

### 2. Receiver 回 DELIVERY_ACK

当服务端收到 `DELIVERY_ACK`：

1. 用 `messageId` 找到 accepted message
2. 校验 `receiverId`
3. 立刻把 SQLite 中该行更新为 `delivered = 1`
4. 更新内存主记录为 delivered 状态
5. 从 `undeliveredMessagesByReceiver[receiverId]` 移除

这样 ACK 后：

- sender 侧幂等主记录仍然保留
- receiver 侧补投索引中已不再包含它
- 服务端重启后也不会再补投这条消息

### 3. 服务端重启

`MessageRouter` 启动时会从 SQLite 恢复：

- 所有 accepted message -> `acceptedMessagesById`
- 所有 `delivered = 0` 的消息 -> `undeliveredMessagesByReceiver`

因此服务端重启后：

- sender 再次发相同 `messageId`，仍然能拿到原始 ACK
- receiver 重新 `AUTH` 后，仍然能拿到尚未 delivery 的消息补投

## 宕机场景说明

### 场景 A：已写库，尚未转发给 B，服务端宕机

结果：

- 消息仍在 SQLite 中
- `delivered = 0`
- 服务重启后恢复进 undelivered bucket
- B 重新 `AUTH` 后可收到补投

### 场景 B：已转发给 B，但 B 尚未回 DELIVERY_ACK，服务端宕机

结果：

- 消息仍在 SQLite 中
- 仍然是 `delivered = 0`
- 服务重启后 B 重新 `AUTH` 时会再次收到
- Android 侧按 `messageId` 去重，因此重复投递是安全的

### 场景 C：B 已回 DELIVERY_ACK，服务端宕机

结果：

- SQLite 中已标记 `delivered = 1`
- 重启后这条消息不会再进入 undelivered bucket
- 不会被再次补投

## 当前限制

B5.5 解决的是 restart-proof accepted message 和 delivery recovery，不是完整历史服务能力。

当前仍有限制：

- 还没有对外提供 B4 服务端历史查询接口
- 当前表结构主要为 accepted-message 恢复与 redelivery 服务
- `ack_json` / `message_json` 直接存协议快照，简单可用，但不是最终历史查询最优结构
- 目前仍然依赖 receiver 后续重新 `AUTH` 才触发补投

## 测试覆盖

当前 mock-server 测试已覆盖：

- accepted message 写入 SQLite
- 服务端重启后 accepted message 恢复
- 服务端重启后 undelivered message 恢复
- receiver `AUTH` 后补投恢复
- 已 `DELIVERY_ACK` 的消息重启后不再补投
- sender 同 `messageId` 跨重启仍幂等

验证命令：

```bash
cd mock-server
mvn -q -Dtest=MessageRouterTest test
mvn -q test
```
