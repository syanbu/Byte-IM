# 项目 C · 自研 IM 客户端需求说明

## 一、项目背景

本项目要求实现一个支持单聊，或进一步支持群聊的 IM 客户端。项目重点不在于直接接入成熟 IM 云服务，而是要求从客户端协议设计、长连接管理、消息可靠性、消息有序性、本地持久化到 UI 展示等核心链路进行自研实现。

项目明确禁止使用任何现成 IM SDK，包括但不限于：

- 火山 RTC IM
- 环信 IM
- 融云 IM
- 腾讯云 IM
- 其他同类现成 IM SDK

服务端可以使用 AI 快速搭建，主要用于辅助完成登录注册、WebSocket 长连接、消息转发、ACK 确认、历史消息查询等能力。服务端不是重点考察部分，重点考察 Android 客户端的协议设计、状态管理、数据库设计、长连接管理与消息收发体验。

## 二、功能需求

### 2.1 基础功能

| 编号 | 功能 | 关键约束 | 说明 |
|---|---|---|---|
| B1 | 登录 / 注册 | HTTP 接口，JWT Token | 用户通过 HTTP 接口完成注册和登录，登录成功后获取 JWT Token，后续 HTTP 请求和 WebSocket 鉴权均基于 Token。 |
| B2 | 单聊文本消息 | WebSocket 长连接，实时收发 | 客户端与服务端建立 WebSocket 长连接，实现单聊文本消息的实时发送与接收。 |
| B3 | 会话列表 | 显示最近会话、未读数、最后一条消息预览 | 首页展示最近会话列表，每个会话需要显示对方信息、最后一条消息、最后消息时间和未读数。 |
| B4 | 历史消息分页拉取 | 上拉加载更多 | 聊天页支持分页加载历史消息，用户上拉时继续加载更早的消息。 |
| B5 | 消息持久化 | 直接使用 SQLite，禁用 Room | 所有消息、会话状态、未读数等数据需要持久化到本地 SQLite，必须使用 `SQLiteOpenHelper` 手写数据库逻辑，禁止使用 Room。 |

---

### 2.2 进阶功能

| 编号 | 功能 | 关键约束 | 说明 |
|---|---|---|---|
| B6 | 自定义二进制协议 | Header + Body + CRC | 自定义协议帧结构，Header 至少包含 `magic`、`version`、`length`、`cmd` 等字段，Body 可使用 Protobuf 或 JSON，协议尾部需要包含 CRC 校验。 |
| B7 | 心跳与断线重连 | 指数退避，前后台切换感知 | 客户端需要定期发送心跳包，检测连接存活状态；断线后使用指数退避策略重连；App 前后台切换时需要调整连接策略。 |
| B8 | 消息有序性 | 客户端生成 seq，服务端确认，乱序重排 | 客户端发送消息时生成本地 seq，服务端返回确认；接收端需要处理乱序消息，并保证最终展示顺序正确。 |
| B9 | 消息可靠性 | ACK 机制，失败重发，去重 messageId | 消息发送后需要等待服务端 ACK；超时或失败时支持重发；客户端和服务端均需要基于 `messageId` 做幂等去重。 |
| B10 | 群聊 + @ 提醒 | 群成员列表，@ 人高亮，@ 我未读单独计数 | 如实现群聊，需要支持群成员列表、@ 指定成员、@ 文本高亮，以及“@ 我”的单独未读提醒计数。 |
| B11 | 图片消息 | 图片上传，渐进式加载 | 支持发送图片消息，图片应先上传到服务端或对象存储，再通过消息体发送图片 URL；聊天页需要支持图片占位、加载中、加载失败和渐进式展示。 |
| B12 | 消息撤回 / 已读回执 | 2 分钟内可撤回，已读状态同步 | 支持发送方在 2 分钟内撤回消息；支持消息已读状态同步，并在 UI 中展示已读 / 未读状态。 |
| B13 | 推送 | 后台被杀时通过厂商推送通道唤起，可用 mock | App 后台或进程被杀时，可以通过厂商推送通道唤起；如真实厂商推送接入成本较高，可使用 mock 方案模拟。 |

## 三、工具与技术选型要求

| 类别 | 必选方案 | 备选方案 | 说明 |
|---|---|---|---|
| 开发语言 | Kotlin | — | Android 客户端必须使用 Kotlin 开发。 |
| 长连接 | OkHttp WebSocket 或裸 Socket + 自写协议帧 | — | 可以选择 OkHttp WebSocket 快速实现长连接，也可以基于裸 Socket 自行封装协议帧。 |
| 数据库 | SQLiteOpenHelper 手写 | — | 禁止使用 Room，需要直接操作 SQLite。 |
| 异步模型 | Coroutines + Flow + Channel | — | 网络收发、数据库读写、UI 状态流转应基于 Kotlin 协程体系。 |
| UI | Jetpack Compose | XML | 优先使用 Jetpack Compose 实现登录页、会话列表页、聊天页等核心界面。 |
| 抓包工具 | Charles / Wireshark | — | 必须抓取自己的协议包，并提交截图和协议字段说明。 |

## 四、验收标准

| 项目 | 标准 | 验收方式 |
|---|---|---|
| 消息延迟 | 同网环境收发小于 200ms | 两台设备或模拟器在同一局域网下测试消息收发延迟。 |
| 消息有序 | 100 条并发发送，接收端 100% 顺序正确 | 编写测试入口或脚本连续发送 100 条消息，接收端按 seq 校验顺序。 |
| 重连成功率 | 模拟 50 次断网，重连成功率 100% | 反复关闭网络 / 切换网络 / 重启服务端连接，记录重连结果。 |
| 数据库性能 | 1 万条消息，会话列表加载小于 200ms | 本地插入 1 万条消息后，测试会话列表查询和渲染耗时。 |
| 协议截图 | 提交 Wireshark / Charles 抓包截图 + 协议字段说明 | 抓取登录、心跳、发送消息、ACK 等协议包，并标注字段含义。 |

## 六、建议开发优先级

### 第一阶段：基础 IM 闭环

目标是先跑通最小可用 IM 链路。

优先完成：

1. B1 登录 / 注册
2. B2 单聊文本消息
3. B3 会话列表
4. B4 历史消息分页拉取
5. B5 SQLite 消息持久化

完成后应达到：

- 用户可以登录。
- 两个用户可以实时收发文本消息。
- 消息可以落库。
- 会话列表可以显示最近聊天。
- 进入聊天页后可以分页查看历史消息。

---

### 第二阶段：协议与可靠性

目标是让项目体现“自研 IM 客户端”的核心技术含量。

优先完成：

1. B6 自定义二进制协议
2. B7 心跳与断线重连
3. B8 消息有序性
4. B9 消息可靠性

完成后应达到：

- 所有 WebSocket 消息统一走自定义协议帧。
- 客户端具备心跳保活能力。
- 断线后可以自动重连。
- 消息发送失败后可以重发。
- 接收端可以处理乱序消息。
- 重复消息不会被重复插入数据库或重复展示。

---

### 第三阶段：体验增强

目标是提升完整度和产品体验。

可选完成：

1. B10 群聊 + @ 提醒
2. B11 图片消息
3. B12 消息撤回 / 已读回执
4. B13 推送 mock

完成后应达到：

- 支持更多消息类型。
- 支持群聊场景。
- 支持撤回、已读等常见 IM 能力。
- 支持后台推送或 mock 推送流程。

## 八、项目重点总结

本项目的核心不是简单做一个聊天界面，而是要体现 IM 客户端的底层能力。

重点包括：

- 自研协议，而不是直接使用现成 IM SDK。
- 自己维护长连接生命周期。
- 自己处理心跳、重连、ACK、重发、去重和有序性。
- 自己设计 SQLite 表结构并完成消息持久化。
- 自己实现会话列表、历史消息分页、未读数等 IM 常见能力。
- 能通过抓包截图说明协议字段，证明协议是真实运行的。

因此，开发时应优先保证基础链路稳定，再逐步补充群聊、图片、撤回、已读和推送等增强能力。
