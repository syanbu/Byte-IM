# B5.5 Mock Server Message Persistence Status

## Requirement

Persist accepted mock-server messages so server restarts do not lose accepted chat data.

## Status

Deferred.

## Background

Current Android B5 persistence stores messages locally with SQLiteOpenHelper. The mock server currently persists users and per-conversation sequence state, but it does not persist accepted chat messages as durable server-side history.

Current mock-server behavior:

- `SEND_MESSAGE` is accepted in memory.
- The server assigns a per-conversation `serverSeq`.
- The server sends `MESSAGE_ACK` to the sender.
- If the receiver is online, the server forwards `RECEIVE_MESSAGE`.
- If the receiver is offline, the server queues the message in an in-memory queue and delivers it after receiver auth.

This means:

- Offline queued messages survive temporary receiver disconnects only while the mock-server process stays alive.
- Offline queued messages are lost if the mock-server process restarts.
- The mock server cannot provide durable server-backed history yet.

## Future Scope

- Add a mock-server SQLite message store.
- Persist each accepted `SEND_MESSAGE` before or atomically with assigning `serverSeq`.
- Store at least:
  - `messageId`
  - `conversationId`
  - `senderId`
  - `receiverId`
  - `clientSeq`
  - `serverSeq`
  - `content`
  - `timestamp` / `serverTime`
  - delivery status metadata if later needed
- Keep `messageId` unique on the server.
- Keep `(conversationId, serverSeq)` unique on the server.
- Restore per-conversation sequence state from persisted messages if the dedicated sequence table is missing or behind.
- Use persisted messages as the foundation for later server-backed history and offline delivery recovery.

## Non-Scope For B9 First Pass

- Do not block B9 sender-side retry on this work.
- Do not implement full history query/backfill here unless explicitly scheduled.
- Do not add receiver delivery ACK here; that belongs to B9.5.

## Verification To Add Later

- Repeated `SEND_MESSAGE` with the same `messageId` does not create duplicate server rows.
- Server restart preserves accepted messages.
- Server restart preserves or reconstructs the next `serverSeq`.
- Offline receiver queued messages can be recovered after server restart.
- Android can receive persisted offline messages after reconnect/auth when that delivery path is implemented.
