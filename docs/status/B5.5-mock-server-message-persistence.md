# B5.5 Mock Server Message Persistence Status

## Requirement

Persist accepted mock-server chat messages and receiver-delivery state so server restart does not lose:

- accepted-message records
- sender-side `messageId` idempotency
- receiver-side undelivered redelivery state
- the storage foundation needed for later B4 server-backed history

## Status

Done for the current mock-server scope.

## Implemented Design

The mock server now persists accepted messages in SQLite at `mock-server/data/mock-im-messages.sqlite`.

Runtime shape:

- `acceptedMessagesById` remains the in-memory primary lookup by `messageId`.
- `undeliveredMessagesByReceiver` remains the in-memory secondary index by `receiverUserId`.
- Both structures are now hydrated from SQLite when `MessageRouter` starts.
- Delivery updates write through to SQLite so restart state matches in-memory state.

Stored fields include:

- `messageId`
- `conversationId`
- `senderId`
- `receiverId`
- `clientSeq`
- `serverSeq`
- `content`
- original sender `timestamp`
- server `serverTime`
- receiver `delivered` flag
- serialized `MESSAGE_ACK` body
- serialized `RECEIVE_MESSAGE` body

## Completed

- Added `SQLiteAcceptedMessageStore` in the mock server.
- `SEND_MESSAGE` now persists the accepted message before it is exposed as the source of truth for restart recovery.
- `MessageRouter` restores accepted messages from SQLite during startup.
- `MessageRouter` rebuilds the undelivered receiver buckets from persisted `delivered = 0` rows during startup.
- Receiver `DELIVERY_ACK` now updates the persisted delivery flag as well as the in-memory undelivered index.
- Receiver auth/reconnect now replays persisted-undelivered messages even after server restart.
- Sender duplicate send with the same `messageId` now remains idempotent after restart: the server returns the original `MESSAGE_ACK`, does not allocate a new `serverSeq`, and does not forward a duplicate `RECEIVE_MESSAGE`.
- The real `MockImServer` startup path now wires `mock-im-messages.sqlite` alongside the existing `mock-im-sequences.sqlite`.

## Preserved Semantics

- `MESSAGE_ACK` still means only "server accepted the message and assigned `serverSeq`".
- `DELIVERY_ACK` still means only "receiver client persisted the message"; it is not a read receipt.
- Conversation-local `serverSeq` monotonicity still depends on the existing `SQLiteServerSeqStore`; B5.5 does not change that contract.
- Redelivery still happens on receiver auth/reconnect, not on sender duplicate send.

## Explicit Non-Scope

- No B4 `HISTORY_QUERY`/`HISTORY_RESULT` API wiring yet.
- No B10-B13 expansion.
- No Android local database refactor.
- No change to sender retry policy, read receipts, or UI semantics.
