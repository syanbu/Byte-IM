# B8 Message Ordering Status

## Requirement

Client generates sequence values, server confirms, and receive side handles out-of-order messages.

## Status

Done for the current B8 scope.

## Sequence Semantics

`clientSeq` is generated by the Android client per conversation when sending a local message.
It means "this client's local send order within this conversation".
It is used by the sender to correlate `MESSAGE_ACK` with the local outgoing message and can describe that sender client's own local send order.
It is not a receiver-side display ordering key, because different senders can produce the same `clientSeq`, local clocks and counters are not authoritative, and multiple connections cannot express the server's final receive order.

`serverSeq` is generated by the mock server per conversation when it accepts `SEND_MESSAGE`.
It means "the server-assigned order in which messages entered this conversation".
It is the authoritative ordering key for confirmed or received messages in a conversation.
Different conversations can reuse the same `serverSeq` values, but the same conversation must increase monotonically.

For single chat, `single:A:B` and `single:A:C` are independent conversations with independent `serverSeq` sequences.

## Completed

- Added `MessageOrderingPolicy` as the shared Android message ordering rule.
- `MessageDao.queryPage()` now orders conversation messages by:
  - local `SENDING` messages with no `serverSeq` first, using `createdAt`, `clientSeq`, and `messageId` as temporary newest-first order;
  - messages with `serverSeq` by `serverSeq DESC` for the internal newest-first list used by the current `reverseLayout` chat UI;
  - legacy or local messages without `serverSeq` and not `SENDING` by local newest-first fallback.
- `AndroidMessageDao` uses the same ordering policy in SQLite `ORDER BY`, without Room.
- `ChatViewModel` merge/refresh logic uses `MessageOrderingPolicy`, so out-of-order `RECEIVE_MESSAGE` arrival is re-sorted after persistence and refresh.
- Android now starts a login-session scoped `MessagePacketProcessor` from the authenticated root, so `RECEIVE_MESSAGE` packets are persisted even when the user is on `Contacts`, `Me`, or another screen instead of `Messages`/`Chat`.
- Duplicate `messageId` handling remains through `insertOrIgnore`; duplicate RECEIVE packets do not create duplicate rows.
- The mock server now allocates `serverSeq` per `conversationId` instead of using one global counter.
- The real mock server persists per-conversation `serverSeq` in `mock-server/data/mock-im-sequences.sqlite`, so restarting the server does not reset a conversation back to `1001` and push new messages behind older local Android rows.
- `MESSAGE_ACK` and `RECEIVE_MESSAGE` bodies carry the assigned `serverSeq`.
- Mock-server logs now include `conversationId`, `clientSeq`, `serverSeq`, sender, receiver, and `messageId` on the send/ack/forward path.

## Explicit Non-Scope

- No ACK timeout retry.
- No retry loop or pending outbox worker.
- No reconnect pending-message replay.
- No gap buffer, receive wait window, or missing-message pull.
- No B10-B13 expansion.

## Remaining Risks

- The current Android chat state remains newest-first internally because `ChatScreen` uses `LazyColumn(reverseLayout = true)`; visually this displays increasing `serverSeq` from older to newer.
- B8 does not implement server-backed gap detection or missing-message sync, so dropped packets still require later B9/sync work.
- Manual emulator verification with deliberately delayed/out-of-order network delivery has not been captured; coverage is currently unit-test based.
- SQLite ordering is implemented in `AndroidMessageDao`, while the local JVM contract test covers the shared in-memory ordering policy. Instrumented SQLite verification is still a useful future check.
