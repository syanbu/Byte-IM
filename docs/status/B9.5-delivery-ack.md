# B9.5 Receiver Delivery ACK Status

## Requirement

Add receiver-side delivery acknowledgement so the server knows a forwarded message was actually received and persisted by the recipient client.

## Status

Done for the current in-memory mock-server scope.

## Background

B9 first pass should focus on sender-side reliability:

- Android pending outbox.
- ACK timeout retry for sender `SEND_MESSAGE`.
- Reconnect-triggered pending resend.
- `FAILED` after retry exhaustion.
- Server-side `messageId` idempotency for repeated sender sends.

That sender `MESSAGE_ACK` only means the server accepted the message and assigned `serverSeq`. It does not prove the receiver client has received or persisted the message.

Current receive path:

```text
A -> Server: SEND_MESSAGE
Server -> A: MESSAGE_ACK
Server -> B: RECEIVE_MESSAGE
```

If the server thinks B is online and writes `RECEIVE_MESSAGE` to the WebSocket, but B disconnects, crashes, or fails before persisting the packet, the server currently has no receiver-side proof.

## Implemented Protocol Shape

Added a receiver delivery ACK command:

```text
Server -> Receiver: RECEIVE_MESSAGE(messageId, conversationId, serverSeq, ...)
Receiver -> Server: DELIVERY_ACK(messageId, conversationId, serverSeq, receiverId)
```

Implemented semantics:

- Android sends delivery ACK only after `RECEIVE_MESSAGE` has been decoded and persisted with `insertOrIgnore`.
- Duplicate `RECEIVE_MESSAGE` packets remain safe because Android deduplicates by `messageId`.
- Server records receiver delivery state only after `DELIVERY_ACK`.
- If `DELIVERY_ACK` is missing, the current mock server redelivers after receiver auth/reconnect within the same server process.
- Redelivery must not create duplicate local rows or duplicate conversation previews.

## Completed

- Added protocol command `DELIVERY_ACK` on Android and mock server.
- Android `MessageRepository` now sends `DELIVERY_ACK` after incoming message persistence, including duplicate `RECEIVE_MESSAGE` cases that are already locally deduplicated.
- Mock server now tracks receiver delivery state per accepted `messageId`.
- Mock server treats duplicate `DELIVERY_ACK` as idempotent.
- Mock server redelivers undelivered messages for the authenticating receiver after auth/reconnect.
- Mock server keeps an in-memory receiver index for undelivered messages: `receiverUserId -> undelivered accepted messages`. Receiver auth now scans only that user's bucket instead of filtering the whole accepted-message table.
- Sender-side `MESSAGE_ACK` semantics remain unchanged: it still means server acceptance only.
- Android receive-path ownership is now explicit: `MessagePacketProcessor` is the single login-session scoped consumer of WebSocket incoming packets.
- `ChatViewModel` and `ConversationListViewModel` no longer call `repository.handlePacket()` from direct `incomingPackets` collection. They refresh from repository/storage updates only.
- This removes the duplicate-consumption path where the same `RECEIVE_MESSAGE` packet could trigger more than one client-side `DELIVERY_ACK` even though local SQLite still deduplicated the `messageId`.

## Server Responsibilities

- Track delivery state per `(messageId, receiverId)`.
- Keep undelivered-message lookup indexed by `receiverId`, so reconnect/auth redelivery targets that receiver's bucket directly.
- Treat duplicate `DELIVERY_ACK` as idempotent.
- Redeliver undelivered messages after receiver reconnect/auth.
- Avoid assigning a new `serverSeq` for redelivery.

## Android Responsibilities

- Persist `RECEIVE_MESSAGE` first.
- Then send delivery ACK.
- If delivery ACK send fails, keep local message; server redelivery later is safe because of `messageId` deduplication.
- Do not expose delivery ACK as a read receipt. Delivery ACK only means "receiver device persisted this message", not "user read it".

## Relationship To Later Features

- Delivery ACK is not the same as sender `MESSAGE_ACK`.
- Delivery ACK is not read receipt; B12 can define read receipts separately.
- Delivery ACK now builds on B5.5 mock-server message persistence, so undelivered receiver replay survives mock-server restart as well as reconnect within one process.

## Explicit Non-Scope

- Do not implement read receipts here.
- Do not change sender-side pending outbox, retry, or `FAILED` semantics from B9.
- Do not change the persisted accepted-message schema into full history APIs here; B4 server-backed history remains separate follow-up work.

## Verification

- Receiver persists a `RECEIVE_MESSAGE` and sends `DELIVERY_ACK`.
- Duplicate `RECEIVE_MESSAGE` produces at most one local row and can still ACK idempotently.
- Server records delivery after ACK.
- Server redelivers an unacked message after receiver reconnect/auth.
- Server does not redeliver an acked message.
